저도 FDE가 하는 일만 들었을 땐 'SI 업무하는거 아닌가?' 라는 생각이 들었는데, 이번 밋업을 다녀온 후 조금은 명확해졌습니다.

## ❓ FDE는 무엇인가

FDE가 실제로 어떻게 움직이는지에 대한 이야기였습니다.

### FDE의 본질 : "의사결정자를 고객에게 가장 가까이"

단순히 '엔지니어가 고객사에 파견되어 세팅을 도와주는 사람' 정도로 이해하면 반쯤은 맞을 것 같습니다.
핵심은 **제품·기술적 의사결정을 내릴 수 있는 사람**이 고객 옆에 있어야 한다는 것.
그래야 커스터마이징, 트레이드오프, 아키텍처 선택이 고객 현장에서 실시간으로 이뤄질 수 있기 때문입니다.

### FDE의 목적은 "밸류"

그래서 FDE의 목적이 뭐냐 물으면, 밸류(Value)라고 정의할 수 있습니다. 그런데 밸류를 정의하는 방법은 **회사마다, 제품마다** 다릅니다.
FDE는 **고객의 프로젝트 본질**을 이해하고, **고객의 데이터 플로우**를 이해하고, 거기서 밸류를 만들어낼 지점을 찾는 것이라고 생각합니다.

### FDE 모션의 장/단점

장점은 명확합니다. **제품을 고객에게 오버핏**시킬 수 있고, 그만큼 **빠르게 움직일 수 있다**는 것.
다만 단점도 그만큼 큽니다. 고객사마다 직접 붙어야 하니 **코스트가 크고**, 그 과정에서
**번아웃**이 오기 쉬우며, 고객사별 요구가 누적되다 보면 **제품의 방향성이 흩어지기** 쉽습니다.

세션에서 반복해서 강조된 얘기가 있습니다. FDE 모션이 살아남으려면 **제품 자체가 커스터마이징
가능한 인프라로 설계**되어 있어야 한다는 것. 그렇지 않으면 고객사마다 커스텀이 쌓이다가
제품이 무너진다고 합니다.

## 💭 마무리

FDE는 '고객 대면 엔지니어링'이라는 겉모습 아래에서 **누가 밸류를 정의하고, 누가 의사결정하고, 어떤 인프라 위에서 작동하는가**라는
질문을 끊임없이 던지는 역할에 가까워 보였습니다.
Claude Code 같은 도구는 이 질문을 훨씬 빠르게 반복할 수 있게 해주는 역할이고요.

밋업 오기 전에 가장 궁금했던 질문이 **'FDE는 SI랑 뭐가 다른 거지?'** 였는데,
이제 누가 물어본다면 **의사결정자 옆에서 제품을 함께 만들어가는 사람.** 이라고 답할 수 있을 것 같습니다.
