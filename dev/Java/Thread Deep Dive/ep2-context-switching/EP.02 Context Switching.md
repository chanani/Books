지난 편에서 "CPU 코어 수보다 훨씬 많은 스레드가 어떻게 동시에 돌아가는 것처럼 보이는가"라는 질문을 던졌습니다.
정답은 **운영체제가 스레드를 매우 빠르게 번갈아 실행하기 때문**인데, 이 전환 과정을 **Context Switching**이라고 합니다.🙂

이 글은 Thread Deep Dive 시리즈의 두 번째 편입니다.

## 시분할(Time-sharing)부터 시작하기

CPU 코어 1개가 1초 동안 처리할 수 있는 명령어는 수십억 개입니다. 사람이 체감하는 시간 단위(수십 ms)에서 보면 CPU는 **거의 무한한 시간** 을 가지고 있습니다.

OS는 이 시간을 잘게 쪼개 여러 스레드에게 나눠줍니다.

```
실제 시간 (수십 ms 단위로 잘라서 본 모습)

CPU Core 1: │ T1 │ T2 │ T3 │ T1 │ T4 │ T2 │ T1 │ T3 │ ...
            └──┬─┴──┬─┴──┬─┴──┬─┴──┬─┴──┬─┴──┬─┴────┘
               └─ 각 칸이 약 10~100ms (Time Slice / Quantum)

사용자 입장에서는 T1~T4가 모두 "동시에" 돌아가는 것처럼 보임
```

이렇게 짧은 시간씩 번갈아 실행하기 위해서는, 스레드가 잠시 멈췄다가 다시 깨어났을 때 **이전에 어디까지 했는지 정확히 기억**할 수 있어야 합니다.
그 기억해야 할 정보가 바로 **컨텍스트(Context)** 이고, 그 정보를 저장하고 복원하는 작업이 **Context Switching**입니다.

---

## Context Switching이란

> **Context Switching:** 현재 실행 중인 스레드의 상태를 저장하고, 다음에 실행할 스레드의 상태를 복원하여 CPU를 넘겨주는 작업.

```
시간 →

  Thread A 실행 중
    │
    │  ① A의 상태 저장 (CPU 레지스터 → A의 TCB)
    ▼
  ┌────────────────────┐
  │  Context Switching │  ← 이 시간 동안 CPU는 "유저 코드"를 실행하지 않음
  │     (커널 작업)      │     (즉, 순수한 오버헤드)
  └────────────────────┘
    │
    │  ② B의 상태 복원 (B의 TCB → CPU 레지스터)
    ▼
  Thread B 실행 시작
```

**핵심:** Context Switching이 일어나는 동안 CPU는 사용자 코드를 한 줄도 실행하지 않습니다. 즉, **순수한 오버헤드**입니다.

---

## 컨텍스트에는 무엇이 담기는가

스레드가 다시 깨어났을 때 마치 멈춘 적이 없던 것처럼 동작하려면, 다음 정보가 모두 보존되어야 합니다.

### 저장 대상

```
스레드의 컨텍스트
  ├─ Program Counter (PC)      : 다음에 실행할 명령어 주소
  ├─ Stack Pointer (SP)        : 현재 스택의 top
  ├─ 범용 레지스터들               : EAX, EBX, ... (x86 기준)
  ├─ 상태 레지스터 (FLAGS)        : 비교 결과, 인터럽트 플래그 등
  ├─ 부동소수점 레지스터            : XMM, YMM 등 (필요 시)
  └─ (프로세스 전환 시) 메모리 맵    : Page Table 베이스 주소
```

이 정보들은 OS가 **TCB(Thread Control Block)** 라는 자료구조에 보관합니다. 프로세스 단위에서는 **PCB(Process Control Block)** 가 같은 역할을 합니다.

```
┌──────────────────────┐
│    TCB of Thread A   │
│  ┌─────────────────┐ │
│  │ Thread ID       │ │
│  │ State           │ │
│  │ PC, SP, Regs    │ │  ← 컨텍스트가 여기 저장됨
│  │ Priority        │ │
│  │ Pointer to PCB  │ │  ← 어느 프로세스에 속하는지
│  └─────────────────┘ │
└──────────────────────┘
```

## User Mode와 Kernel Mode

Context Switching을 이해하려면 먼저 CPU의 두 가지 실행 모드를 알아야 합니다.

```
┌─────────────────────────────────┐
│        User Mode (Ring 3)       │  ← 일반 애플리케이션 코드
│   - 제한된 명령어                  │     Java 애플리케이션 등
│   - 메모리 접근 제한                │
└─────────────────────────────────┘
              ↕  System Call / Interrupt
┌─────────────────────────────────┐
│      Kernel Mode (Ring 0)       │  ← OS 커널 코드
│   - 모든 명령어 실행 가능            │    스케줄러, 디바이스 드라이버
│   - 모든 메모리 접근 가능            │
└─────────────────────────────────┘
```

스레드를 전환하는 작업(스케줄링, 레지스터 저장/복원)은 **반드시 Kernel Mode에서** 이루어집니다. 사용자 코드가 임의로 다른 스레드의 상태를 건드릴 수 있다면 보안과 안정성이 무너지기 때문입니다.

### Mode Switch도 비용이다

User Mode ↔ Kernel Mode 전환 자체에도 비용이 있습니다. 이를 **Mode Switch**라고 합니다.

```
Mode Switch ⊆ Context Switch (반드시는 아니지만 거의 항상 동반됨)

  User Mode (Java 코드 실행 중)
        │
        │ 인터럽트 / 시스템 콜
        ▼
  Kernel Mode 진입 (Mode Switch)
        │
        │ 스케줄러가 다음 스레드 결정
        ▼
  Context Switch (TCB 저장/복원)
        │
        │ 새 스레드의 컨텍스트로 복원
        ▼
  User Mode 복귀 (Mode Switch)
        │
        ▼
  새 스레드의 Java 코드 실행
```

## 프로세스 vs 스레드 — Context Switching 비용 차이

이게 멀티스레드를 쓰는 가장 큰 이유 중 하나입니다.

### 프로세스 간 Context Switching (비싸다)

```
Process A → Process B로 전환

  ① A의 레지스터 저장
  ② B의 레지스터 복원
  ③ 메모리 맵(Page Table) 전환  ← 이게 비쌈
       └─ TLB(Translation Lookaside Buffer) 무효화
       └─ CPU 캐시도 거의 무효화됨 (다른 프로세스의 메모리니까)
```

**TLB**는 가상 주소 → 물리 주소 매핑을 캐시해두는 하드웨어입니다. 프로세스가 바뀌면 메모리 맵이 통째로 바뀌므로 TLB를 비워야(flush) 합니다. 이후 한동안은 메모리 접근마다 페이지 테이블을 직접 뒤져야 하므로 **간접 비용**이 큽니다.

### 스레드 간 Context Switching (싸다)

```
Thread A → Thread B로 전환 (같은 프로세스 내)

  ① A의 레지스터 저장
  ② B의 레지스터 복원
  ③ 메모리 맵 그대로 유지 ✓     ← 같은 프로세스니까 공유
       └─ TLB 그대로 유효
       └─ 캐시도 상당 부분 유효
```

같은 프로세스 안의 스레드들은 **메모리 맵을 공유**하므로 Page Table 전환이 필요 없습니다. TLB도 그대로 쓸 수 있고, CPU 캐시도 일부는 살아있을 가능성이 높습니다.

### 비용 비교

| 항목 | 프로세스 간 | 스레드 간 |
|---|---|---|
| 레지스터 저장/복원 | O | O |
| 메모리 맵 전환 | O (비쌈) | X |
| TLB Flush | O | X |
| 캐시 무효화 | 거의 전부 | 일부만 |
| **상대적 비용** | **높음** | **낮음** |

> 💡 그래서 멀티스레드가 멀티프로세스보다 가볍다고 하는 겁니다. 단, 메모리를 공유하기 때문에 **동기화 문제** 라는 새로운 비용이 생깁니다(EP.04에서 다룹니다).

## Context Switching이 발생하는 시점

OS가 언제 스레드를 전환할지 결정하는 트리거는 크게 4가지입니다.

```
┌─────────────────────────────────────────────────────────┐
│         Context Switching이 발생하는 4가지 상황              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ① Time Slice 만료 (Preemptive)                          │
│     스레드가 할당된 시간을 다 씀                               │
│                                                         │
│  ② I/O 블로킹 (Voluntary)                                │
│     디스크/네트워크 read, DB 쿼리 등                         │
│     → 스레드가 스스로 BLOCKED/WAITING으로 전환                │
│                                                         │
│  ③ 우선순위 더 높은 스레드 등장 (Preemptive)                  │
│     인터럽트, 더 급한 작업이 RUNNABLE이 됨                     │
│                                                         │
│  ④ 자발적 양보 (Voluntary)                                │
│     Thread.yield(), Thread.sleep(), Object.wait()       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

Java에서 자주 마주치는 시나리오를 코드로 보면:

```java
// ② I/O 블로킹
String content = Files.readString(path);   // 디스크 I/O 동안 다른 스레드에 양보

// ④ 자발적 양보
Thread.sleep(100);                         // 명시적으로 양보
synchronized (lock) {                      
    lock.wait();                           // 락을 풀고 양보
}

// 그리고 BlockingQueue 같은 동시성 자료구조도 내부적으로 ④에 해당
queue.take();                              // 큐가 비어있으면 양보
```

JVM 시리즈 EP.02에서 본 스레드 상태(`BLOCKED`, `WAITING`, `TIMED_WAITING`)들이 모두 이 시점에 발생합니다.

## Context Switching의 실제 비용

> 그래서 한 번에 얼마나 비싼가?

리눅스 환경에서 측정한 일반적인 수치입니다.

```
직접 비용 (Direct Cost)
  ├─ 레지스터 저장/복원        : ~1 µs
  ├─ Mode Switch              : ~1~2 µs
  └─ 스케줄러 실행              : ~1~2 µs
                                ─────
                                약 3~5 µs

간접 비용 (Indirect Cost) — 이게 더 큽니다
  ├─ CPU 캐시 무효화           : 수십~수백 µs
  ├─ TLB miss 누적 비용        : 수십 µs
  └─ 분기 예측 정보 손실        : 측정 어려움

→ 종합하면 한 번의 컨텍스트 스위칭은 대략 수~수십 µs 수준
→ 초당 수만 번 일어나면 무시할 수 없는 오버헤드
```

CPU가 1ns에 한 명령어를 실행한다고 치면, 5µs는 **명령어 5,000개분의 시간**을 그냥 날리는 셈입니다.

### 모니터링하는 방법

리눅스에서는 `vmstat`이나 `pidstat`로 확인할 수 있습니다.

```bash
$ vmstat 1
procs ----memory---- ---swap-- -----io---- --system-- ---cpu---
 r  b  swpd  free   ...        in    cs    us sy id wa
 2  0     0  ...               5234  12456  20  3 76  1
                                     ─────
                                     초당 컨텍스트 스위칭 횟수 (cs)
```

`cs` 값이 비정상적으로 높다면(수십만/초 이상) 스레드가 너무 많거나 락 경합이 심한 신호입니다.

## 스레드는 많을수록 좋은가?

학습 초반에 가장 빠지기 쉬운 함정입니다.

```
잘못된 직관:
  스레드 100개로 처리하면 빠르겠지!
  → 스레드 1000개로 늘리면 더 빠르겠지!
  → 10000개면 최고겠지!

실제로는:
  스레드 수가 일정 지점을 넘으면
  Context Switching 오버헤드가 처리량을 갉아먹기 시작.
  처리량이 오히려 감소함.
```

```
처리량
  │           ╱─────╲
  │         ╱        ╲────────  ← 스레드 늘려도 효과 없음
  │       ╱                     실제로는 감소함
  │     ╱
  │   ╱
  │ ╱
  └────────────────────────── 스레드 수
        ↑
        sweet spot
        (CPU 코어 수 + α)
```

### 적정 스레드 수의 경험적 공식

워크로드 특성에 따라 다릅니다.

| 워크로드 | 권장 스레드 수 |
|---|---|
| **CPU 집약적** (계산, 압축, 암호화) | CPU 코어 수 + 1 |
| **I/O 집약적** (DB, 외부 API) | 코어 수 × (1 + 대기시간/계산시간) |
| **혼합** | 측정해서 튜닝 |

Spring Boot의 Tomcat 기본값(`server.tomcat.threads.max=200`)은 I/O 집약적인 일반 웹 요청을 가정한 수치입니다.

## Virtual Thread가 등장한 이유

이 지점에서 자연스럽게 다음 의문이 생깁니다.

> "스레드가 비싸서 풀로 재사용해야 하고, 너무 많이 만들면 Context Switching이 폭발한다.
> 그런데 Java 21의 Virtual Thread는 수십만 개를 만들어도 된다고? 어떻게?"

핵심은 **Virtual Thread는 OS 스레드가 아니다** 라는 점입니다.

```
일반 스레드 (Platform Thread)
  Java Thread → OS Thread (1:1)
  → Context Switching = OS 커널 작업 = 비쌈

Virtual Thread
  Java Thread (수만 개) → 캐리어 스레드 (수십 개) → OS Thread
  → 가상 스레드 간 전환은 JVM이 사용자 모드에서 처리
  → Mode Switch 없음, 메모리 맵 전환 없음
  → 매우 저렴
```

이 부분은 **EP.03 JVM의 스레드 모델**에서 자세히 다룹니다. 거기서 1:1 / N:1 / M:N 매핑 모델을 보고 나면 Virtual Thread가 왜 가벼운지 명확해집니다.

이번 편에서 다룬 내용을 정리하면 다음과 같습니다.

| 주제 | 핵심 내용 |
|---|---|
| Context Switching | 스레드 전환 시 상태(Context)를 저장/복원하는 작업 |
| 저장 대상 | PC, SP, 레지스터, FLAGS, (프로세스면) 메모리 맵 |
| User/Kernel Mode | 스케줄링은 반드시 Kernel Mode |
| 프로세스 vs 스레드 비용 | 스레드 전환이 압도적으로 저렴 (메모리 맵 공유) |
| 발생 시점 | 시간 만료 / I/O / 우선순위 / 자발적 양보 |
| 비용 | 직접 ~5µs + 캐시·TLB 무효화의 간접 비용 |
| 적정 스레드 수 | CPU 집약적이면 코어 수 + 1, I/O 집약적이면 더 많이 |

이제 OS가 스레드를 어떻게 전환하는지는 알게 됐습니다.
그런데 우리는 Java 개발자입니다. 다음 질문은 자연스럽게 이거죠.

> JVM의 `Thread` 객체와 OS의 스레드는 어떻게 연결되는가?
> JVM은 어떤 스레드 모델을 쓰는가? 1:1? N:1? M:N?

다음 편에서는 **JVM의 스레드 모델** 을 살펴봅니다. JVM 시리즈에서 다룬 JVM Stack과 PC Register가 OS 스레드와 어떻게 매핑되는지, 그리고 Java 21의 Virtual Thread가 왜 게임 체인저인지 본격적으로 알아보겠습니다.
