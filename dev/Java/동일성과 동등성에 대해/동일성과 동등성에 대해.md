동일성과 동등서엥 대해 쉽게 넘어갈 수 있는 개념이지만,
잘못 이해하고 있으면 시간을 한참을 날려버릴 수 있기에 한번 정리해보려 합니다.

## ⭐️ 동일성과 동등성이란?

### 동일성 (Identity)

동일성은 두 객체가 **완전히 같은 객체인지**를 판단합니다.
즉, 메모리 상에서 같은 주소를 참조하고 있는지 여부입니다. Java에서는 `==` 연산자가 이 역할을 합니다.

### 동등성 (Equality)

동등성은 두 객체가 **논리적으로 같은 값을 가지고 있는지**를 판단합니다.
Java에서는 `equals()` 메서드를 통해 확인하며, `Object` 클래스의 기본 구현은 `==`와 동일하게 동작합니다.
그렇기 때문에 "값이 같다"는 의미로 사용하려면 반드시 재정의가 필요합니다.

## 🔎 실제 코드로 살펴보기

### String에서의 함정

가장 흔하게 마주치는 사례는 문자열 비교입니다.

```java
String a = "hello";
String b = "hello";
String c = new String("hello");

System.out.println(a == b);        // true
System.out.println(a == c);        // false
System.out.println(a.equals(c));   // true
```

`a`와 `b`는 **String Pool**에 저장된 같은 리터럴을 참조하기 때문에 `==`가 true를 반환합니다.
하지만 `new` 키워드로 생성된 `c`는 Heap 영역에 새 객체로 할당되기 때문에 `==`로 비교하면 false가 나옵니다.

### Integer Cache

더 까다로운 케이스는 `Integer`입니다.

```java
Integer a = 127;
Integer b = 127;
Integer c = 128;
Integer d = 128;

System.out.println(a == b); // true
System.out.println(c == d); // false
```

Java는 `-128 ~ 127` 범위의 `Integer` 값을 캐싱해두기 때문에 `a`와 `b`는 같은 객체를 참조합니다.
하지만 `128`부터는 캐시 범위를 벗어나 새 객체가 생성되므로 `==` 비교가 실패합니다.
**Wrapper 타입을 `==`로 비교하면 안 되는 이유**가 여기 있습니다.

## 🛠️ equals()와 hashCode() 재정의하기

### 왜 함께 재정의해야 할까?

`equals()`만 재정의하고 `hashCode()`를 방치하면 `HashMap`, `HashSet` 같은 해시 기반 컬렉션에서 심각한 버그가 발생합니다.
논리적으로 같은 객체임에도 해시값이 달라 서로 다른 버킷에 저장되기 때문입니다.

```java
Set<Member> members = new HashSet<>();
members.add(new Member("홍길동", 25));
members.add(new Member("홍길동", 25));

System.out.println(members.size()); // hashCode 재정의 안 하면 2
```

Java의 규약은 명확합니다.

> `equals()`가 true를 반환하는 두 객체는 반드시 같은 `hashCode()`를 가져야 한다.

### equals 규약

`equals()`를 재정의할 때는 다음 규약을 반드시 지켜야 합니다.

- **반사성**: `x.equals(x)`는 항상 true
- **대칭성**: `x.equals(y)`가 true면 `y.equals(x)`도 true
- **추이성**: `x.equals(y)`, `y.equals(z)`가 true면 `x.equals(z)`도 true
- **일관성**: 객체가 변경되지 않는 한 결과가 일정해야 함
- **null 비교**: `x.equals(null)`은 항상 false

### 그래서 record와 Lombok을 쓴다

위 규약을 직접 지키는 것은 생각보다 어렵고 실수하기 쉽습니다.
그래서 Java 16부터 도입된 `record`나 Lombok의 `@EqualsAndHashCode`를 활용해 자동 생성하는 것이 안전합니다.

```java
public record Member(String name, int age) {}
```

이 한 줄로 `equals()`, `hashCode()`, `toString()`이 모두 규약에 맞게 구현됩니다.

## 💭 마치며

처음 Java를 배웠을 때는 `==`와 `equals()`의 차이를 그냥 "참조 비교냐 값 비교냐" 정도로만 알고 있었습니다.
하지만 실무에서 `Set`에 중복이 쌓이는 버그를 마주하고, 테스트가 이유 모르게 실패하는 경험을 하면서
**동일성과 동등성은 단순한 비교 연산의 차이가 아니라 객체의 정체성을 정의하는 문제**라는 것을 깨달았습니다.

객체를 설계할 때 "이 객체는 무엇으로 같다고 판단할 것인가?"라는 질문은 곧 도메인에 대한 이해와 직결됩니다.
`Member`를 식별자로 비교할지, 이름과 나이로 비교할지에 따라 코드의 의미가 완전히 달라지기 때문입니다.
